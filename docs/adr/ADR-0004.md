# ADR-0004 · Estructura hexagonal en capas bajo `src/`

| Campo          | Valor                                                              |
|----------------|--------------------------------------------------------------------|
| Estado         | Aceptada · 26-06-2026                                             |
| Contexto       | El cómputo del riesgo es la funcionalidad crítica y debe ser defendible ante el jurado: testeable sin levantar framework ni base de datos, y reutilizable desde ETL y API. A la vez, la app creció en sub-sistemas (authnocion, perfil de proveedor, fraccionamiento) que conviene aislar. |
| Decisión       | Organizar `src/` en cuatro capas con dependencias dirigidas hacia adentro: `domain/` (entidades, motor de riesgo, validadores — cero deps externas), `application/` (casos de uso + puertos `ObrasRepository` / `UsersRepository` / `CryptoPort`), `infrastructure/` (adaptadores: `pg.Pool`, repositorios Postgres/JSON, captcha-store y firma de sesiones con `node:crypto`), y `app/` + `components/` + `lib/` (presentación Next.js App Router). El dominio no importa Next, `pg` ni React. |
| Consecuencias  | **+** Tests viven al lado del dominio (`*.test.ts`) y corren sin DB.<br>**+** La ETL (`etl/`) puede importar `domain/` para validar banderas (ya espejado en `etl_ocds.mjs`).<br>**+** Cambiar de Postgres a otra fuente solo requiere otro adaptador del puerto `ObrasRepository`.<br>**−** Duplicación temporal de armado de `AuthDeps` en `auth/login` y `auth/register` (deuda marcada en código).<br>**−** El almacenamiento de usuarios/sesiones hoy es `globalThis` in-memory; migrar a Postgres exige añadir tablas `usuario`/`sesion`/`consulta_log` no presentes aún en `schema.sql`. |

## Capas y reglas de dependencia

```
src/
├── domain/          ← entidades, risk-engine, proveedor-risk,
│                      fraccionamiento, validators (puro, + tests)
├── application/     ← use-cases.ts, auth-use-cases.ts, ports.ts
│                      (depende solo de domain + interfaces)
├── infrastructure/  ← db/ (client.ts, schema.sql),
│                      repositories/ (index factory, JSON/PG adapters),
│                      auth/ (captcha-store.ts, session.ts)
├── app/             ← Next.js App Router:
│                      layout/page (landing), plataforma/, login/,
│                      registro/, buscar-proveedor/, api/ (route handlers)
├── components/      ← UI cliente Leaflet: MapaObras, VistaCiudadano,
│                      VistaProveedor, PerfilRiesgo, Captcha, ObraCard, Info
└── lib/            ← format.ts (helpers de presentación)
```

Regla: una flecha solo puede apuntar dePresentation → Application → Domain. Infrastructure implementa los puertos declarados en Application. Domain no conoce a nadie.

## Puertos actuales (contratos de `application/ports.ts` y `auth-use-cases.ts`)

- `ObrasRepository`: `listarRegiones`, `buscarPorRegion`, `obtenerObra`, `buscarContratosPorRuc`.
- `UsersRepository` / `SessionsRepository` (aún en memoria, pendiente de adaptador Postgres).
- `CryptoPort`: hash/verify scrypt + firmar/verificar sesión HMAC-SHA256.
- `CaptchaVerifier`: valida `"<token>:<respuesta>"` contra `captcha-store`.
